# CloudCDN — パフォーマンス

## エッジネットワーク
CloudCDN は Cloudflare のグローバルネットワーク上に構築されています:
- **300 以上のデータセンター**が 100 以上の国、193 以上の都市にあります。
- **世界人口の 95%** が Cloudflare PoP から 50ms 以内に位置しています。
- **8,000 以上のネットワーク相互接続**による最適なルーティング。
- Anycast ルーティングがリクエストを自動的に最寄りの PoP へ送ります。

## レイテンシ
| リージョン | 中央値 TTFB(キャッシュヒット) | P95 TTFB |
|----------|------------------------------|---------|
| 北米 | 30ms 未満 | 80ms 未満 |
| ヨーロッパ | 35ms 未満 | 90ms 未満 |
| アジア太平洋 | 50ms 未満 | 120ms 未満 |
| 南米 | 60ms 未満 | 150ms 未満 |
| アフリカ | 80ms 未満 | 200ms 未満 |

これらはエッジキャッシュヒット時の値です。PoP への最初のリクエストはオリジン取得が必要で、一度だけ 50〜200ms 追加されます。

## キャッシュ戦略
すべてのアセットで強力な不変キャッシュを使用します:
```
Cache-Control: public, max-age=31536000, immutable
```

これは次のことを意味します:
- **ブラウザキャッシュ:** 1 年。再検証リクエストなし。
- **CDN エッジキャッシュ:** 1 年。最寄りの PoP から配信。
- **キャッシュヒット率:** 本番アセットで 95% 以上。
- **キャッシュ無効化:** 更新したアセットを配信するにはファイル名やパスを変更します。

manifest.json は短命のキャッシュを使用します:
```
Cache-Control: public, max-age=300
```

## 画像圧縮

### 形式比較
| 形式 | 典型的なサイズ(PNG 比) | ブラウザサポート | 用途 |
|------|----------------------|----------------|------|
| PNG | ベース(100%) | 100% | 可逆、透過 |
| WebP | PNG の約 40% | 97% | 一般的な Web 配信 |
| AVIF | PNG の約 30% | 93% | 最大圧縮 |
| SVG | 可変 | 100% | ベクターグラフィック、アイコン |

### 自動変換の設定
- **WebP:** 品質 80、非可逆。品質とサイズの最適なバランス。
- **AVIF:** 品質 65、非可逆。良好な視覚的忠実度を維持しながら最大圧縮。
- 元ファイル(PNG/JPEG)は生成バリアントとともに常に保持されます。

### 実際の節約効果
50 個のアイコン(16x16〜512x512)を持つ典型的なプロジェクトの場合:
- PNG 合計: 約 5 MB
- WebP 合計: 約 2 MB(60% 削減)
- AVIF 合計: 約 1.5 MB(70% 削減)

バナー画像(1200x630)の場合:
- PNG: 平均約 500 KB
- WebP: 平均約 150 KB
- AVIF: 平均約 90 KB

## Core Web Vitals への影響
CloudCDN を通じて最適化された画像を配信することで、次が直接改善されます:

### LCP(Largest Contentful Paint)
- 目標: 2.5 秒未満。
- 効果: PNG の代わりに AVIF を配信することで、画像の多いページでは LCP を 40〜60% 削減できます。
- ヒント: ファーストビューの画像は `<link rel="preload" as="image">` でプリロードします。

### CLS(Cumulative Layout Shift)
- 目標: 0.1 未満。
- 効果: `<img>` タグには常に `width` と `height` 属性を設定して領域を確保します。
- ヒント: レスポンシブレイアウト計算には manifest.json の `size` フィールドを使用します。

### INP(Interaction to Next Paint)
- 目標: 200ms 未満。
- 効果: 画像が小さいほどメインスレッドのデコード作業が減ります。
- ヒント: ファーストビュー外の画像には `loading="lazy"` を使用して初期ページサイズを削減します。

## デプロイパフォーマンス
| メトリック | 値 |
|----------|-----|
| 増分デプロイ(1〜10 ファイル変更) | 5〜15 秒 |
| 完全デプロイ(10,000 以上のファイル) | 30〜60 秒 |
| 画像圧縮(PNG 1 枚あたり、CI) | 約 200ms |
| マニフェスト生成(10,000 以上のアセット) | 5 秒未満 |

デプロイはコンテンツハッシュによる重複排除を使用し、新規または変更されたファイルのみがアップロードされます。最初のデプロイ以降、後続のプッシュでは通常変更されたファイルのみがアップロードされます。

## モニタリング
- **Cloudflare Analytics:** Cloudflare ダッシュボード → Workers & Pages → cloudcdn-pro → Metrics で利用可能。
- **GitHub Actions:** ビルドおよびデプロイログは Actions タブで確認できます。
- **ステータス:** ホームページ(cloudcdn.pro)が稼働状況を表示します。

## 最適化のヒント
1. **可能な限り AVIF URL を使用してください** — PNG より 70% 軽量です。
2. **形式フォールバックには `<picture>` 要素を使用してください**(AVIF → WebP → PNG)。
3. ファーストビュー上の**重要な画像はプリロード**してください。
4. ファーストビュー外のすべてを **`loading="lazy"` で遅延読み込み**してください。
5. レイアウトシフトを防ぐため、すべての `<img>` タグに**明示的なサイズを設定**してください。
6. **必要最小限のアイコンサイズを使用してください** — 64x64 で十分なときに 512x512 を配信しないでください。
7. **Pro プラン:** `?format=auto` を使って CloudCDN に最適な形式を自動配信させてください。
